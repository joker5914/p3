# P3 Firestore — Collection Schema & Setup Guide

This document is the authoritative reference for the Firestore collections used by P3. **Only three collections are used.** The old `imported_data` collection should be deleted.

---

## Collections Overview

| Collection | Purpose | Written by | Read by |
|---|---|---|---|
| `plates` | Plate registration registry | Backend import / add_plate.py | Backend on every scan |
| `plate_scans` | Scan event log | Backend `/api/v1/scan` | Backend dashboard & history |
| `school_admins` | User roles, status, invite tracking | Backend `/api/v1/users/invite` | Backend auth middleware |

---

## `plates` — Plate Registry

**Document ID:** HMAC-SHA256 of the plate number (keyed with `SECRET_KEY`). Never the raw plate number.

```
plates/{plate_token}
  student_names_encrypted : string | string[]   Fernet-encrypted student name(s)
  parent                  : string              Fernet-encrypted guardian name
  guardian_id_encrypted   : string              Fernet-encrypted guardian email
  school_id               : string              Plain — used for access scoping
  vehicle_make            : string | null
  vehicle_model           : string | null
  vehicle_color           : string | null
  imported_at             : string (ISO 8601)
```

**Written by:** `POST /api/v1/admin/import-plates` and `add_plate.py` CLI.

**Read by:** `POST /api/v1/scan` — looks up the plate token to get student/guardian names.

**Delete:** `DELETE /api/v1/plates/{plate_token}` — school_admin only.

---

## `plate_scans` — Scan Event Log

**Document ID:** Auto-generated by Firestore.

```
plate_scans/{auto_id}
  plate_token             : string              Same token as the plates doc ID
  student_names_encrypted : string | string[]   Fernet-encrypted (copied from plates)
  parent_name_encrypted   : string              Fernet-encrypted (copied from plates)
  timestamp               : timestamp           Local device time (timezone-aware)
  location                : string              Scanner location label
  confidence_score        : number (0.0–1.0)    OCR confidence from RPi scanner
  hash                    : string              HMAC-SHA256 integrity hash
  school_id               : string              Access scoping
```

**Written by:** `POST /api/v1/scan` — every plate detection from the RPi scanner.

**Read by:**
- `GET /api/v1/dashboard` — today's queue (ascending timestamp)
- `GET /api/v1/history` — paginated history with date/name filters
- `GET /api/v1/reports/summary` — statistics

**Delete:** `DELETE /api/v1/scans/clear` — clears all scans for the school (school_admin only).

### Required Firestore Indexes

These composite indexes are defined in `firestore.indexes.json` and deployed automatically via `firebase deploy --only firestore:indexes`:

| Collection | Fields | Order | Used by |
|---|---|---|---|
| `plate_scans` | `school_id` ASC, `timestamp` ASC | Dashboard query |
| `plate_scans` | `school_id` ASC, `timestamp` DESC | History query |
| `plates` | `school_id` ASC, `imported_at` DESC | Plates list |
| `school_admins` | `school_id` ASC, `display_name` ASC | Users list |

## `school_admins` — User Management

**Document ID:** Firebase Auth UID.

```
school_admins/{uid}
  uid             : string              Firebase Auth UID
  email           : string              Plain — not sensitive, used for display
  display_name    : string
  school_id       : string              Access scoping
  role            : "school_admin" | "staff"
  status          : "pending" | "active" | "disabled"
  invited_by_uid  : string              UID of the admin who created this account
  invited_at      : timestamp
  created_at      : timestamp
```

**Written by:**
- `POST /api/v1/users/invite` — creates the record with `status: pending`
- `GET /api/v1/me` — transitions `pending` → `active` on first login
- `PATCH /api/v1/users/{uid}/role` — updates role field
- `PATCH /api/v1/users/{uid}/status` — updates status field
- `Backend/firestore_setup.py` — one-time bootstrap for the first admin

**Read by:**
- `verify_firebase_token()` — checks role & status on every authenticated request
- `GET /api/v1/users` — lists all users for the school
- `GET /api/v1/me` — returns the calling user's profile

**Role definitions:**

| Role | Can do |
|---|---|
| `school_admin` | Everything — import plates, invite users, clear scans, delete plates |
| `staff` | View dashboard, view history, dismiss individual queue entries |

## What to delete: `imported_data`

The `imported_data` collection was created by the old `/api/v1/admin/import` endpoint which has been removed. It stored raw/incorrectly-encrypted plate data and is no longer read or written by any part of the system.

**Delete it:** Firebase Console → Firestore → `imported_data` → ⋮ → **Delete collection**.

## First-time setup checklist

```
[ ] 1. Delete the imported_data collection in Firebase Console
[ ] 2. Deploy security rules:  firebase deploy --only firestore:rules
[ ] 3. Deploy indexes:         firebase deploy --only firestore:indexes
[ ] 4. Bootstrap your admin account:
        python Backend/firestore_setup.py \
          --uid  <your Firebase UID> \
          --email you@example.com \
          --name  "Your Name" \
          --school your_school_id
[ ] 5. Log out and back into the admin portal
[ ] 6. Import plates via the Data Importer or add_plate.py CLI
[ ] 7. Verify the plates collection has documents with plate_token as the doc ID
```

## How the security rules work

`firestore.rules` enforces these guarantees client-side (the Admin SDK used by the backend bypasses them):

- **Authentication required** on all paths — unauthenticated requests are rejected.
- **School scoping** — every read/write checks that `resource.data.school_id` matches the caller's `school_id` custom claim. Users from one school cannot access another school's data even if they know the document IDs.
- **Role enforcement** — mutations to `plates`, `school_admins`, and `plate_scans` (delete) require the `school_admin` role in the custom claim.
- **Client writes to `plate_scans` are blocked** — only the backend service account (Admin SDK) can create scan records. This prevents clients from injecting fake scan events.
- **Catch-all deny** — any path not explicitly listed (including the deleted `imported_data`) returns denied.

## Deploying rules and indexes

```bash
# From the repo root
firebase deploy --only firestore:rules
firebase deploy --only firestore:indexes

# Or both at once
firebase deploy --only firestore
```

Make sure your `firebase.json` includes:

```json
{
  "firestore": {
    "rules": "firestore.rules",
    "indexes": "firestore.indexes.json"
  },
  "hosting": { ... }
}
```
